Pourquoi ce design ?

Un établissement peut vouloir Ollama (local, RGPD). Une PME préfère Groq (rapide, gratuit). Un lycée utilise Gemini Flash. On s'adapte sans refactoriser.

Défaut de production

Ollama + llama3.2:3b — 100 % local, ~2 Go RAM, zéro coût, zéro donnée hors UE. Configurable via Admin Panel sans redémarrage.

llm/services/ · docs/02-llm-integration.md

Fallback J4

En cas de surcharge Ollama → bascule automatique vers Groq ou Gemini. Configuré dans le panel Admin. Stratégie de scalabilité J4.

Pourquoi pas C ou D ?

Gemini et Groq : excellente latence, mais les données de cours quittent l'UE. Invalide notre argument RGPD B2B. Non négociable.

Pourquoi B ?

100 % local, 2 Go RAM, bascule en < 1h — une seule ligne .env. Différenciateur préservé. Meilleur compromis vitesse/qualité/RGPD.

J3 · Mercredi — Perturbation Sécurité LLM

Le LLM peut être manipulé. On ferme la faille.

Prompt injection : un utilisateur peut envoyer "Ignore les instructions précédentes et..." dans le champ cours. Détecté, patchés, testés.

Correctifs appliqués

  • Troncature texte à 4 000 chars côté backend
  • Validation correct_index : entier 0-3 seulement
  • Limite max_tokens sur la sortie LLM
  • Distribution forcée des bonnes réponses
  • Sanitisation du contenu cours avant injection prompt

llm/views.py · quizzes/serializers.py

5 tests adversariaux

  • Injection directe
  • Lorem ipsum (hors-sujet)
  • Texte vide / trop court
  • JSON malformé
  • Exfiltration de prompt

tests/test_adversarial.py

5/5
Attaques bloquées

Erreur 400 ou réponse vide contrôlée — jamais de contenu injecté dans la sortie finale.

Principe retenu

Le LLM est une entrée non fiable comme toute autre API externe. On sanitise en amont ET on valide la sortie en aval. Defense in depth.
